\begin{abstract}
Federated learning servers must aggregate client updates without observing client data, making them vulnerable to both untargeted model poisoning and targeted backdoor attacks. Existing robust aggregators mainly rely on update geometry, coordinate-wise robust statistics, or alignment with a single trusted reference direction. These signals can be brittle when benign clients are heterogeneous and adaptive attackers remain close to the benign update manifold. We present \textbf{CARAT}, a class-aware robust aggregation rule based on \emph{hidden task certificates}. CARAT uses server-visible labeled reference data to instantiate multiple hidden, class-balanced micro-tasks and scores each client update by the loss decrease it induces on those tasks. It then combines robust norm clipping, common-radius evaluation, worst-task optimization through a soft minimum over certificate scores, coordinate-rank anomaly penalties, and capped-simplex weight optimization with temporal regularization. This design favors updates that are broadly useful across semantic sub-tasks while suppressing clients that remain structurally extreme in rank space. In a verified FedAvg/CIFAR100/FangAttack run, CARAT substantially improves over Mean and assigns zero weight to the attacker coalition in late rounds. Because the strongest completed CARAT run currently available disables test-time evaluation, this evidence should be read as an initial empirical validation rather than a complete benchmark; nevertheless, it supports the promise of task-aware aggregation and motivates completion of the clean-accuracy and ASR benchmark now underway.
\end{abstract}

\section{Introduction}

Federated learning (FL) enables many clients to train a shared model without centralizing their raw data \citep{mcmahan2017fedavg}. This decentralization is attractive for privacy, governance, and communication, but it also leaves the server in a fundamentally asymmetric position: it must update the global model using client messages that it cannot directly verify. Poisoning attacks exploit exactly this asymmetry. Malicious clients can slow or prevent convergence, bias the learned decision boundary, or implant triggers that activate attacker-chosen behavior at test time \citep{fang2020local,shejwalkar2021manipulating,bagdasaryan2020backdoor,wang2020tails,zhang2022neurotoxin}. In practice, the security of FL is therefore tightly coupled to the aggregation rule.

The central difficulty is that malicious behavior and benign heterogeneity can look superficially similar. Under non-i.i.d.\ partitions, benign clients legitimately differ in direction, norm, sparsity, and class emphasis. As a result, an update can appear unusual either because it is harmful or because it reflects a different local data distribution. Many existing defenses rely on signals that are too coarse to resolve this ambiguity reliably. Pairwise distances and geometric centrality can misclassify benign dispersion as adversarial deviation; coordinate-wise robust statistics suppress only marginal extremes and ignore cross-coordinate structure; trusted-reference schemes reduce the server's auxiliary information to a single global direction, which can miss failures concentrated on particular classes or semantic regions.

This observation motivates the main question behind CARAT: \emph{can the server evaluate an update by testing whether it is consistently useful across hidden semantic sub-tasks, rather than only by testing whether it looks similar to other updates?} We answer this question with \textbf{CARAT}, \emph{Class-Aware Robust Aggregation via Hidden Task Certificates}. CARAT assumes server-side access to labeled reference data and uses that data to sample hidden, class-balanced micro-tasks. Each client update is then scored by the loss reduction it induces across these tasks. The resulting certificate matrix gives the server a finer-grained notion of utility: an update should be trusted not merely because it looks central, but because it remains broadly helpful across multiple hidden slices of the learning problem.

CARAT complements this task-aware signal with two additional ingredients. First, it uses robust clipping and common-radius evaluation so that certificate comparisons are not dominated by raw magnitude. Second, it adds a coordinate-rank anomaly penalty to capture whether a client remains structurally extreme even after clipping. Aggregation weights are then learned on a capped simplex through a worst-task objective with temporal regularization. The overall goal is to preserve benign diversity while suppressing updates that are either semantically brittle or structurally anomalous.

This design places CARAT between trust-based aggregation and task-aware validation. Compared with FLTrust, it expands trusted data into many hidden probes rather than a single trusted direction. Compared with heterogeneity-aware schemes such as bucketing, it does not try to homogenize clients before aggregation; instead, it directly asks whether each update is useful across hidden task slices. Compared with inspection-oriented defenses such as FLAME and DeepSight, it intervenes at aggregation time rather than primarily through post-hoc filtering. In this sense, CARAT does not merely introduce another distance rule. It changes the server's validation signal from update similarity to hidden-task utility.

\paragraph{Contributions.}
Our contributions are as follows.
\begin{itemize}
\item We introduce hidden-task certificates as a task-aware server-side validation signal for robust FL aggregation, using multiple class-balanced micro-tasks instead of a single trusted reference direction.
\item We formulate CARAT as a capped-simplex weight optimization problem that jointly enforces worst-task utility, rank-space anomaly suppression, and temporal regularization.
\item We make the implementation assumptions explicit, most importantly the server-side access to labeled data from which the hidden-task probe pool is constructed.
\item We report the current repository-backed empirical evidence and separate verified results from the larger clean-accuracy and ASR benchmark suite that is still being completed.
\end{itemize}

\section{Method}

\subsection{Problem setting}

We consider synchronous FL with $n$ clients and a global parameter vector $w_r \in \R^d$ at round $r$. At each round, client $i$ returns a message $u_i^r$, which the server maps to a vector-form update $\Delta_i^r \in \R^d$. For FedAvg, $\Delta_i^r$ can be viewed as the client model delta relative to the current global model; for gradient-style optimizers, it is the client update itself. The server must aggregate $\{\Delta_i^r\}_{i=1}^n$ into a single global step while a fraction $\rho$ of clients may be malicious and may coordinate their behavior.

\paragraph{Server knowledge.}
CARAT assumes the server has access to labeled reference data. From this reference set it builds a probe pool and repeatedly samples hidden, class-balanced validation tasks. This is a materially stronger assumption than fully data-free robust aggregation and should be treated as part of the method definition rather than as an incidental implementation detail.

\paragraph{Goal.}
The goal is to assign aggregation weights that preserve benign diversity while suppressing updates that are either semantically harmful or structurally anomalous. The key challenge is to distinguish malicious deviation from legitimate non-i.i.d.\ variation without relying only on geometric similarity.

\subsection{CARAT overview}

CARAT is organized around three ideas. First, it validates updates through hidden task utility rather than through client-to-client similarity alone. Second, it emphasizes \emph{worst-task} performance, reflecting the intuition that an update should not be trusted if it helps on average but damages a subset of hidden semantic slices. Third, it augments semantic utility with a structural anomaly signal in rank space.

Operationally, each round has three stages. The server first places updates into a fair evaluation geometry by clipping and rescaling them. It then scores each client using hidden-task certificates and coordinate-rank anomaly penalties. Finally, it solves a capped-simplex optimization problem whose solution defines the aggregation weights. A crucial design choice is that normalized updates are used only for \emph{evaluation}, whereas clipped updates are used for the actual model step. CARAT therefore separates evaluation geometry from update geometry.

\subsection{Hidden-task and anomaly scoring}

Let $\mathcal{D}^{\mathrm{ref}}$ denote the server reference data and let $\mathcal{P}$ denote a probe pool sampled from it. At round $r$, CARAT samples $T$ hidden tasks from $\mathcal{P}$, with each task constructed to be approximately class-balanced. This design is the source of CARAT's class-aware behavior: a malicious update may look globally plausible while degrading only a subset of classes, whereas balanced hidden tasks make such class-selective failures easier to detect.

\paragraph{Clipping and common-radius evaluation.}

Before hidden-task evaluation, CARAT robustly clips client updates. Let $\|\Delta_i\|_2$ denote the norm of client $i$'s update. In our implementation, the clipping threshold $\tau_r$ is estimated by a MAD-based rule:
\begin{equation}
\tau_r = \operatorname{median}(\{\|\Delta_i\|_2\}) +
k \cdot 1.4826 \cdot \operatorname{MAD}(\{\|\Delta_i\|_2\}),
\label{eq:clip}
\end{equation}
and the clipped update is
\begin{equation}
\bar{\Delta}_i = \min\left(1, \frac{\tau_r}{\|\Delta_i\|_2 + \varepsilon}\right)\Delta_i.
\end{equation}

CARAT then rescales each clipped update to a common evaluation radius $q_r$:
\begin{equation}
\tilde{\Delta}_i = q_r \frac{\bar{\Delta}_i}{\|\bar{\Delta}_i\|_2 + \varepsilon},
\label{eq:normalize}
\end{equation}
where $q_r$ is chosen as a quantile of the positive clipped norms. This normalization removes pure scale advantages during hidden-task comparison. Importantly, it does \emph{not} define the final server step; it only creates a common evaluation geometry in which client updates can be compared more fairly. Appendix Proposition~\ref{prop:directional-invariance} formalizes the invariance induced by this step.

\paragraph{Hidden-task certificates.}

For each hidden task $t$, let $\ell_t(w)$ denote the average cross-entropy loss of model $w$ on that task. CARAT defines a \emph{hidden-task certificate}
\begin{equation}
c_{i,t} = \ell_t(w_r) - \ell_t(w_r + \tilde{\Delta}_i).
\label{eq:cert}
\end{equation}
Here, ``certificate'' denotes an internal task-level utility score rather than a formal certified-robustness guarantee. Positive values indicate that client $i$ improves task $t$ relative to the current global model.

Stacking these scores yields a certificate matrix $C \in \R^{n \times T}$. Because different tasks may have different intrinsic difficulties and baseline loss scales, CARAT robustly standardizes each column of $C$ before optimization. This preserves within-task ordering while making scores comparable across tasks.

\paragraph{Rank-space anomaly penalties.}

Hidden-task utility alone is not sufficient: an attacker may overfit to the probe tasks or align with a subset of them while still remaining structurally atypical. CARAT therefore augments semantic utility with a coordinate-rank anomaly signal.

For a sampled coordinate set $S \subseteq \{1,\dots,d\}$, CARAT ranks clients coordinate-wise and measures the average normalized distance from the center rank:
\begin{equation}
r_i^{(S)} = \frac{1}{|S|}\sum_{j \in S}
\left|
\frac{\operatorname{rank}_{i,j}}{n-1} - \frac{1}{2}
\right|.
\label{eq:rank}
\end{equation}
CARAT repeats this computation over several random coordinate subsets and aggregates the resulting values robustly. A client receives a large anomaly score if its update remains persistently extreme across many coordinates. After robust standardization, only positive deviations are retained as penalties. The signal is intentionally asymmetric: suspicious extremeness is penalized, whereas unusual centrality is not explicitly rewarded.

\subsection{Weight optimization and aggregation}

CARAT chooses aggregation weights $\alpha \in \R^n$ on the capped simplex
\begin{equation}
\mathcal{A}(\hat{\rho}) =
\left\{
\alpha \in \R^n :
\alpha_i \ge 0,\;
\sum_{i=1}^n \alpha_i = 1,\;
\alpha_i \le \frac{1}{(1-\hat{\rho})n}
\right\},
\label{eq:capped-simplex}
\end{equation}
where $\hat{\rho}$ is an estimate of the attacker fraction. This cap prevents any single client from dominating the aggregate and guarantees a minimum support size. Appendix Proposition~\ref{prop:capped-simplex} states the corresponding feasibility and support guarantees.

Let $r \in \R^n$ denote the positive rank penalties and let $\alpha^{\mathrm{prev}}$ be the previous round's weights. CARAT solves
\begin{equation}
\max_{\alpha \in \mathcal{A}(\hat{\rho})}
\;\softmin_{\beta}(C^\top \alpha)
- \lambda_r r^\top \alpha
- \lambda_p \|\alpha - \alpha^{\mathrm{prev}}\|_2^2,
\label{eq:objective}
\end{equation}
where
\begin{equation}
\softmin_{\beta}(s_1,\dots,s_T)
= -\frac{1}{\beta}\log\left(\frac{1}{T}\sum_{t=1}^T e^{-\beta s_t}\right).
\label{eq:softmin}
\end{equation}
The softmin term gives a smooth approximation to worst-task utility: large $\beta$ places more emphasis on the weakest hidden tasks, while finite $\beta$ preserves optimization stability. The second term penalizes structurally extreme clients, and the third term regularizes the solution toward the previous round to reduce unnecessary oscillation. Appendix Proposition~\ref{prop:softmin} records the basic bounds and gradient form of the softmin term.

Eq.~\eqref{eq:objective} yields a clear selection principle. A client should retain influence only if it is useful across hidden tasks \emph{and} not persistently extreme in rank space. The temporal prior does not define trust by itself; instead, it resolves ambiguity among utility-equivalent feasible solutions and discourages gratuitous weight oscillation. Appendix Proposition~\ref{prop:temporal-prior} formalizes this role.

\paragraph{Final aggregation and complexity.}

After optimization, CARAT aggregates the clipped updates:
\begin{equation}
\Delta_{\mathrm{agg}} = \sum_{i=1}^n \alpha_i \bar{\Delta}_i.
\label{eq:aggregate}
\end{equation}
This yields the final server step. The final update therefore inherits the robustness benefits of clipping while using weights learned from the richer evaluation geometry defined above.

Relative to standard aggregation, the main additional cost comes from evaluating $n$ candidate updates over $T$ hidden tasks and computing rank penalties on coordinate subsamples. Appendix Proposition~\ref{prop:complexity} summarizes the resulting per-round complexity. In practice, the probe-model forward passes dominate, while the rank and optimization stages are lower-order terms with tunable budgets.

\section{Experiments}

\subsection{Experimental setup}

\paragraph{Datasets and models.}
All paper-side experiment specs use FedAvg on CIFAR100 with a ResNet-18 backbone. We intentionally fix the dataset-model pair across the clean-reference, untargeted-poisoning, backdoor, and ablation suites so that changes in robustness can be attributed to the defense rather than to architecture changes. CIFAR100 is particularly relevant for CARAT because the method evaluates client updates across class-balanced hidden tasks; a larger class space makes class-selective failure modes more visible than in low-class-count benchmarks. The strongest completed legacy CARAT pilot also uses the same CIFAR100/ResNet-18 pair, so its evidence is directly comparable to the ongoing benchmark suite.

\paragraph{Attacks and compared defenses.}
We use separate suites for clean, untargeted, and backdoor evaluation. The clean-reference suite uses \texttt{NoAttack} and compares \{Mean, FLTrust, MultiKrum, CARAT\}. The main untargeted suite evaluates \{ALIE, MinMax, FangAttack\} against \{Mean, FLTrust, MultiKrum, CARAT\}. The backdoor suite evaluates \{BadNets, Neurotoxin\} against \{Mean, FLTrust, FLAME, DeepSight, CARAT\}. In addition, the strongest finished legacy pilot available today compares Mean, TriGuardFL, and CARAT under i.i.d.\ FangAttack. We continue to report that pilot because it is currently the only completed CARAT run with end-to-end logs strong enough to anchor a concrete main-text table.

\paragraph{FL setting.}
CARAT is implemented as a server aggregator in FLPoison and operates on the vector-form client deltas produced by the runtime. All committed paper-side specs use synchronous FedAvg with 20 clients, 300 communication rounds, local epoch count $4$, batch size $64$, and learning rate $0.05$. The clean-reference suite uses attacker fraction $0$, the main untargeted suite sweeps attacker fractions $\{0.2,0.4\}$, and the backdoor suite uses attacker fraction $0.2$. Heterogeneity is represented by i.i.d.\ partitions and Dirichlet non-i.i.d.\ splits: \texttt{clean\_reference.yaml} uses $\alpha \in \{0.1,0.5\}$, \texttt{main\_untargeted.yaml} uses $\alpha \in \{0.1,0.5\}$, and \texttt{backdoor.yaml} uses $\alpha=0.5$. Each paper-side suite runs three repeats, yielding effective seeds $42$, $43$, and $44$. The default CARAT hyperparameters are: probe pool size $512$, $8$ hidden tasks, $4$ probe samples per class, probe batch size $128$, probe-radius quantile $0.5$, MAD clipping with $k=2.5$, default attacker prior $\hat{\rho}=0.2$, certificate temperature $\beta=12.0$, rank weight $0.20$, prior weight $0.05$, $4096$ sampled coordinates, $5$ rank subsamples, $60$ optimization steps, and optimizer learning rate $0.5$.

\paragraph{Metrics.}
For clean and untargeted runs, \texttt{metrics\_exp*.csv} records per-round training and test accuracy and loss. For backdoor runs, we report clean test accuracy together with attack success rate (ASR); ASR and related attack-specific diagnostics are parsed from the detailed run logs. To capture computational cost, we also report mean round time and end-to-end runtime when those quantities are available. One important caveat is that the strongest completed legacy CARAT pilot was produced from an earlier base config with \verb|evaluate=false|. Its main-text evidence is therefore limited to training accuracy, training loss, and runtime, whereas all paper-side benchmark suites are configured to record test-time metrics.

The most diagnostic CARAT-specific signal is the behavior of the learned weights. In this benchmark, the coordinator assigns the first \texttt{num\_adv} worker slots to attackers, so clients $0$--$3$ are malicious. Late in training, CARAT repeatedly logs weight vectors of the form
\[
\alpha = [0,0,0,0,0.0625,0.0625,\ldots,0.0625],
\]
which means the optimizer places zero mass on the entire malicious coalition while distributing the remaining mass uniformly over benign clients. At epoch 298, for example, the log records task scores $[0.2879, 0.2784, 0.2859, 0.2781, 0.2533, 0.3009, 0.2930, 0.2876]$, a softmin value of $0.2820$, evaluation radius $18.1484$, and clip norm $18.3227$. These quantities indicate that the implementation is doing more than hard filtering: it is identifying weak hidden tasks under a common scoring radius and then applying the optimized weights to the clipped updates used for aggregation.

These observations are encouraging, but they still do not establish improved clean-test accuracy, reduced ASR, or robustness across non-i.i.d.\ regimes for CARAT. The remaining empirical gap is no longer instrumentation but coverage: the evaluation-enabled CARAT benchmark has started, yet its CARAT rows, non-i.i.d.\ tasks, backdoor suite, and ablations are not complete.

\subsection{Current scope and limitations}

A core methodological limitation remains the server-side data assumption already stated in the Method section: in the current code path, the probe pool is sampled from a server-visible labeled training dataset rather than from a tiny external reference set. This assumption matters when CARAT is compared to fully data-free robust aggregators.

Computational cost is a second practical limitation. Compared with lightweight rules such as Mean, CARAT adds repeated hidden-task forward passes, rank subsampling, and projected optimization. On the current FangAttack/CIFAR100 pilot, the mean round time is $9.49$s for CARAT versus $1.66$s for Mean and $7.37$s for TriGuardFL. Any final empirical case must therefore justify CARAT's extra server-side cost with corresponding clean-test or ASR gains, not only with training-time diagnostics.

Finally, empirical coverage is still incomplete. The strongest completed CARAT evidence is a single-seed, training-only FangAttack pilot. An evaluation-enabled benchmark suite that includes CARAT is now committed and running, but the CARAT rows, non-i.i.d.\ cases, backdoor tasks, and adaptive-attack studies are not yet complete. This means the current manuscript supports a methodology claim and an initial empirical result, but not yet a full benchmark claim.

\section{Related Work}

The most relevant prior work differs primarily in \emph{what signal} it uses to decide whether a client should influence the global model. This perspective is useful for evaluating CARAT because the method does not mainly contribute a new distance metric or a new filter; it changes the unit of validation from client-to-client similarity to client-to-task utility.

\paragraph{Similarity-based robust aggregation.}
Classical Byzantine-robust rules infer benignness from centrality. Krum and related geometry-based methods favor updates that remain close to other clients under pairwise distances \citep{blanchard2017krum,elmhamdi2018bulyan}, while coordinate-wise median and trimmed mean suppress marginal extremes independently \citep{yin2018median}. These methods are appealing because they require little server-side prior knowledge. Their weakness is that they implicitly assume benign updates should look mutually similar. Under non-i.i.d.\ data, that assumption becomes fragile: benign clients may be dispersed for legitimate reasons, and adaptive attackers may remain close to the benign manifold while still harming a subset of classes or semantic regions.

\paragraph{Heterogeneity-aware and reference-data defenses.}
Two important responses to this limitation are to reshape the client population or to inject trusted server knowledge. Bucketing reduces heterogeneity by averaging clients before robust aggregation \citep{karimireddy2022bucketing}. FLTrust instead uses a trusted server dataset to compute a reference direction and reweights clients by alignment with that anchor \citep{cao2021fltrust}. Other defenses, including FoolsGold and FLDetector, add historical or representation-level diagnostics rather than relying purely on instantaneous geometry \citep{fung2020foolsgold,zhang2022fldetector}. CARAT is closest to this family because it also exploits server-side labeled data. The key difference is that it does not compress trusted information into one global direction. Instead, it expands that information into many hidden validation tasks, allowing the server to ask whether an update is broadly useful rather than merely well aligned.

\paragraph{Adaptive poisoning and backdoor defenses.}
Modern attacks are explicitly optimized against robust aggregators. FangAttack, MinMax, MinSum, and ALIE are representative untargeted attacks designed to evade simple geometric or statistical filters \citep{fang2020local,shejwalkar2021manipulating,baruch2019alie}. On the backdoor side, model replacement, edge-case attacks, and durable attacks such as Neurotoxin show that an update can remain outwardly plausible while still encoding highly targeted malicious behavior \citep{bagdasaryan2020backdoor,wang2020tails,zhang2022neurotoxin}. Backdoor-oriented defenses such as FLAME, DeepSight, and CRFL respond with clustering, inspection, clipping, or smoothing mechanisms \citep{nguyen2022flame,rieger2022deepsight,xie2021crfl}. Taken together, this literature suggests that one-dimensional defenses are increasingly brittle: when the server validates only one property, adaptive attackers can often optimize against exactly that property.

\paragraph{Positioning of CARAT.}
CARAT is best understood as a task-aware aggregation rule for the regime where benign heterogeneity and adaptive poisoning interact. Compared with FLTrust, it uses trusted data to instantiate many hidden semantic probes rather than one trust anchor. Compared with bucketing, it does not pre-average clients to reduce dispersion; it directly evaluates each client on hidden task slices. Compared with FLAME and DeepSight, it intervenes at aggregation time instead of relying primarily on post-hoc inspection or clustering. The intended contribution is therefore not another robustness heuristic layered on top of update similarity, but a different validation principle: a useful client update should remain helpful across hidden sub-tasks, not merely look centrally located.

\section{Conclusion}

CARAT combines hidden class-balanced task certificates, rank-space anomaly suppression, and capped-simplex weight optimization into a single aggregation rule. The central claim of the paper is not merely that robust aggregation should filter outliers, but that it should validate client updates against multiple hidden semantic slices of the task. This perspective is what allows CARAT to separate benign diversity from malicious inconsistency more directly than geometry-only or single-reference defenses.

The current repository already supports a concrete methodology claim and an initial empirical claim: in a completed CIFAR100/FangAttack pilot, CARAT suppresses attacker weights and substantially improves over Mean, albeit with higher runtime and without yet surpassing every strong baseline on that single seed. The immediate next step is to complete the evaluation-enabled untargeted, backdoor, and ablation suites so that the final paper reports clean accuracy, ASR, runtime overhead, and multi-seed variance under both i.i.d.\ and non-i.i.d.\ settings.

If those experiments confirm the pilot pattern, CARAT would offer a practically relevant path toward more reliable FL aggregation in settings where class-selective failures matter, including safety-sensitive deployments. Even if the gains turn out to be conditional, the hidden-task view still provides a useful organizing principle for designing and stress-testing future FL defenses.

\section{Theoretical Analysis}

\begin{proposition}[Feasibility and support of the capped simplex]
\label{prop:capped-simplex}
Let
\[
\mathcal{A}(\hat{\rho})=
\left\{
\alpha \in \R^n:
\alpha_i \ge 0,\;
\sum_{i=1}^n \alpha_i = 1,\;
\alpha_i \le \frac{1}{(1-\hat{\rho})n}
\right\},
\]
with $\hat{\rho}<1$. Then $\mathcal{A}(\hat{\rho})$ is non-empty, and every $\alpha \in \mathcal{A}(\hat{\rho})$ satisfies
\[
|\operatorname{supp}(\alpha)| \ge (1-\hat{\rho})n.
\]
\end{proposition}

\begin{proof}[Proof sketch]
Let $u = 1/((1-\hat{\rho})n)$. Since $\hat{\rho}<1$, we have $u \ge 1/n$, so the uniform weight vector is feasible and the set is non-empty. For the support bound, if only $k$ coordinates of $\alpha$ are positive, then
\[
1=\sum_{i=1}^n \alpha_i \le k u = \frac{k}{(1-\hat{\rho})n},
\]
which implies $k \ge (1-\hat{\rho})n$. Therefore CARAT can suppress suspicious clients, but cannot collapse the aggregate onto an arbitrarily small subset of participants.
\end{proof}

\begin{proposition}[Softmin bounds and gradient form]
\label{prop:softmin}
Let $s \in \R^T$ and define
\[
\softmin_{\beta}(s)
= -\frac{1}{\beta}\log\left(\frac{1}{T}\sum_{t=1}^T e^{-\beta s_t}\right).
\]
Then
\[
\min_t s_t \le \softmin_{\beta}(s) \le \frac{1}{T}\sum_{t=1}^T s_t.
\]
Moreover, if $z(\alpha)=C^\top\alpha$ and
\[
\pi_t(\alpha)=\frac{\exp(-\beta z_t(\alpha))}{\sum_{\ell=1}^T \exp(-\beta z_\ell(\alpha))},
\]
then
\[
\nabla_\alpha \softmin_{\beta}(C^\top\alpha)=C\pi(\alpha).
\]
\end{proposition}

\begin{proof}[Proof sketch]
The lower and upper bounds follow from standard log-sum-exp inequalities: the exponential average is bounded below by the largest exponential term and above by the arithmetic mean. Monotonicity in each coordinate is immediate. Differentiating the softmin expression with respect to $z_t$ yields the Gibbs weights $\pi_t(\alpha)$, and the chain rule then gives $C\pi(\alpha)$. Hence the ascent direction is a convex combination of certificate columns, with more weight placed on tasks whose current aggregate utility is smaller.
\end{proof}

\begin{proposition}[Directional invariance of certificate scoring]
\label{prop:directional-invariance}
Suppose two nonzero clipped updates satisfy $\bar{\Delta}_i = a \bar{\Delta}_j$ for some scalar $a>0$. Then their normalized versions in Eq.~\eqref{eq:normalize} coincide, i.e.,
\[
\tilde{\Delta}_i=\tilde{\Delta}_j.
\]
Consequently, their hidden-task certificates are identical:
\[
c_{i,t}=c_{j,t}\qquad \text{for all } t \in \{1,\dots,T\}.
\]
\end{proposition}

\begin{proof}[Proof sketch]
Eq.~\eqref{eq:normalize} rescales every nonzero clipped update to the common radius $q_r$. Positive collinearity is therefore removed by normalization:
\[
q_r\frac{\bar{\Delta}_i}{\|\bar{\Delta}_i\|_2}
=
q_r\frac{a\bar{\Delta}_j}{\|a\bar{\Delta}_j\|_2}
=
q_r\frac{\bar{\Delta}_j}{\|\bar{\Delta}_j\|_2}.
\]
Substituting the resulting equality into Eq.~\eqref{eq:cert} gives identical task scores. Thus, after clipping, the certificate stage depends on direction rather than residual norm magnitude.
\end{proof}

\begin{proposition}[Temporal prior and projected-iterate invariants]
\label{prop:temporal-prior}
Define
\[
f(\alpha)=\softmin_{\beta}(C^\top\alpha)-\lambda_r r^\top\alpha-\lambda_p\|\alpha-\alpha^{\mathrm{prev}}\|_2^2.
\]
If $\alpha,\alpha' \in \mathcal{A}(\hat{\rho})$ satisfy $C^\top\alpha=C^\top\alpha'$ and $r^\top\alpha=r^\top\alpha'$, then
\[
f(\alpha)-f(\alpha')
=
-\lambda_p\!\left(\|\alpha-\alpha^{\mathrm{prev}}\|_2^2-\|\alpha'-\alpha^{\mathrm{prev}}\|_2^2\right).
\]
In addition, under projected gradient ascent
\[
\alpha^{(k+1)}=\Pi_{\mathcal{A}(\hat{\rho})}\!\left(\alpha^{(k)}+\eta \nabla f(\alpha^{(k)})\right),
\]
every iterate remains in $\mathcal{A}(\hat{\rho})$.
\end{proposition}

\begin{proof}[Proof sketch]
The first statement follows by direct substitution: if the certificate and rank terms match, the only remaining difference in objective value is the quadratic prior term. CARAT therefore prefers, among utility-equivalent feasible weights, the solution closer to $\alpha^{\mathrm{prev}}$. The second statement is immediate from the projection operator. Hence the optimizer preserves three round invariants throughout: $\alpha_i \ge 0$, $\sum_i \alpha_i =1$, and $\alpha_i \le 1/((1-\hat{\rho})n)$.
\end{proof}

\begin{proposition}[Per-round complexity]
\label{prop:complexity}
Let $n$ be the number of clients, $T$ the number of hidden tasks, $m$ the number of sampled coordinates for rank scoring, $S$ the number of rank subsamples, and $K$ the number of projected-gradient steps. Ignoring constant factors from batch size and model width, the extra defense-side work of CARAT scales as
\[
\text{probe-model forward passes} + O(Snm + KnT).
\]
\end{proposition}

\begin{proof}[Proof sketch]
CARAT performs one baseline probe evaluation plus $n$ candidate probe evaluations over the $T$ hidden tasks, then computes $S$ rank summaries over $m$ coordinates, and finally executes $K$ projected optimization steps over $n$ weights and $T$ task scores. In the current implementation, the probe-model forward passes dominate wall-clock time, while the rank and optimization stages are lower-order terms controlled by $m$, $S$, and $K$.
\end{proof}

\paragraph{Remark on assumptions.}
The theoretical object $D^{\mathrm{ref}}$ in the main text should be read as the probe pool \emph{after} subsampling. The code, however, obtains that pool by sampling from the full server-visible training dataset. Any final formal discussion should keep this distinction explicit, because it materially strengthens the server-side assumption relative to a fully data-free defense.

\section{Algorithm}

\paragraph{Inputs.}
Current global model $w_r$, client messages $\{u_i^r\}_{i=1}^n$, previous-round weights $\alpha^{\mathrm{prev}}$, server-visible training dataset, and CARAT hyperparameters.

\paragraph{Persistent state.}
CARAT maintains two pieces of state across rounds: a balanced probe pool sampled at initialization from the server-visible training dataset, and the previous-round weight vector $\alpha^{\mathrm{prev}}$, which serves as the temporal prior in the current round.

\paragraph{Initialization.}
At the first round, the implementation initializes $\alpha^{\mathrm{prev}}$ to the uniform vector. This makes the initial round unbiased across clients while still allowing later rounds to exploit temporal consistency once CARAT has observed several optimization trajectories.

\paragraph{Procedure.}
\begin{enumerate}
\item Convert client messages to vector-form deltas $\Delta_i^r$ using the common update-preparation utility.
\item Compute client norms, choose a clipping threshold with the configured MAD or percentile rule, and form clipped deltas $\bar{\Delta}_i^r$.
\item Choose an evaluation radius from the configured quantile of positive clipped norms and normalize clipped deltas to obtain $\tilde{\Delta}_i^r$.
\item Sample class-balanced hidden tasks from the fixed probe pool for round $r$; if strict balanced sampling fails, fall back to replacement and, if necessary, to a random subset rather than aborting the round.
\item Load the current global weights into the server-side probe model, evaluate baseline hidden-task losses, and compute certificate scores $c_{i,t} = \ell_t(w_r) - \ell_t(w_r + \tilde{\Delta}_i^r)$ for all clients and tasks.
\item Robustly standardize certificate columns, compute rank-space penalties from coordinate subsamples of $\bar{\Delta}_i^r$, and retain only positive standardized rank deviations.
\item Resolve $\hat{\rho}$, initialize weights from $\alpha^{\mathrm{prev}}$, and optimize
\[
\max_{\alpha \in \mathcal{A}(\hat{\rho})}
\softmin_{\beta}(C^\top\alpha)-\lambda_r r^\top\alpha-\lambda_p\|\alpha-\alpha^{\mathrm{prev}}\|_2^2
\]
by projected gradient ascent on the capped simplex.
\item Update $\alpha^{\mathrm{prev}}$ with the optimized solution.
\item Aggregate the clipped deltas $\bar{\Delta}_i^r$ with the optimized weights and return the resulting global step.
\end{enumerate}

\paragraph{Round invariants and failure handling.}
Two implementation choices are important for robustness. First, the optimizer never leaves the capped simplex, so every intermediate and final weight vector remains feasible. Second, probe-task construction is fail-soft rather than fail-stop: if strict class-balanced sampling is impossible, the code falls back to replacement or to a random subset, ensuring that the round can still proceed with a well-defined certificate matrix.

\paragraph{Outputs.}
Aggregated server update, updated temporal prior $\alpha^{\mathrm{prev}}$, and diagnostic logs including learned weights, task scores, softmin focus, rank penalties, evaluation radius, clipping threshold, and the identities of the hidden probe tasks used in the current round.
